## Findings

1. High: mutating an issue does not preserve unknown `br` fields exactly, which breaks the project's core compatibility guarantee. [`UpdateRecord`](/Users/annasmith/projects/bt/internal/issue.go#L147) unmarshals the original JSON into `map[string]any` and then re-marshals it at [`internal/issue.go#L179`](/Users/annasmith/projects/bt/internal/issue.go#L179). That loses original key order, whitespace, and any lexeme-level fidelity for unknown fields. The current round-trip tests only cover load/save without mutation, so this regression is untested; see [`internal/test_issue_test.go#L11`](/Users/annasmith/projects/bt/internal/test_issue_test.go#L11) and the absence of any test around `UpdateRecord` preserving foreign fields.

2. High: ID generation does not match the documented/public contract. [`GenerateID`](/Users/annasmith/projects/bt/internal/issue.go#L199) emits 3-character suffixes for small repos and derives them deterministically from title/desc/creator/timestamp, while the repo contract says IDs are always `<prefix>-<4 base36 chars>` and derived from timestamp plus random bytes. The tests actively lock in the incompatible behavior at [`internal/test_issue_test.go#L97`](/Users/annasmith/projects/bt/internal/test_issue_test.go#L97). Since IDs are user-visible and part of br/bd compatibility, this is more than an internal implementation detail.

3. Medium: `dep add` accepts self-dependencies and duplicate edges with no validation. In [`cmd/dep.go#L44`](/Users/annasmith/projects/bt/cmd/dep.go#L44) through [`cmd/dep.go#L88`](/Users/annasmith/projects/bt/cmd/dep.go#L88), the command resolves both IDs and blindly appends a new dependency. A self-edge makes an issue permanently non-ready, and repeated `dep add` calls create duplicate blockers in the file. That is easy for an agent workflow to do accidentally and there is no test covering rejection or deduplication.

4. Medium: `archive` is not atomic across its two-file mutation, so a partial failure can duplicate issues between active and archived storage. [`cmd/archive.go#L76`](/Users/annasmith/projects/bt/cmd/archive.go#L76) appends to `archive.jsonl` first, then [`cmd/archive.go#L94`](/Users/annasmith/projects/bt/cmd/archive.go#L94) rewrites `issues.jsonl`. If the second step fails or the process dies in between, the same issue exists in both files. Given the repo explicitly calls out atomic writes for JSONL mutation, this is a real integrity gap.

## Open Questions / Assumptions

- I am assuming the AGENTS contract is authoritative for compatibility requirements. If the intended behavior changed, the docs and tests need to be updated together because they currently disagree with the implementation on IDs and round-tripping.
- I did not find a current failing test; `go test ./...` passes. The bigger issue is that the suite is missing coverage for the highest-risk compatibility paths.
