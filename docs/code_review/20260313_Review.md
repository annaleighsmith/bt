# Code Review: bt Issue Tracker

Review of the `bt` codebase for correctness, safety, and maintainability.

## Findings

### 1. Divergence from Documented ID and Dependency Model
- **What's wrong:** `docs/ID_AND_CONCURRENCY.md` specifies that epic subtasks should use `<parent>.<n>` format (e.g., `bt-a1b2.1`). However, `cmd/create.go` (via the `--parent` flag) generates a standard top-level ID for all issues. Additionally, `docs/ENUMS_AND_FORMATS.md` states that epic/subtask relationships should use the `parent-child` dependency type, but both `create --parent` and `dep add` hardcode the dependency type as `blocks`.
- **Why it matters:** This creates an inconsistency between the tool's behavior and its documentation. It also makes it impossible to distinguish between a hard dependency (blocking) and a structural relationship (parent/child) as intended by the design.
- **Where:** `cmd/create.go` (lines 99-105), `cmd/dep.go` (lines 75-83).
- **Severity:** High

### 2. Weak ID Resolution Logic
- **What's wrong:** `internal.ResolveID` uses `strings.HasSuffix(id, fragment)` as its first check. For example, if fragments `1` is provided, it will match any ID ending in `1` (e.g., `bt-a1`, `bt-b1`, `bt-c11`). If multiple matches exist, it correctly errors as ambiguous, but the suffix match is far too broad and likely to cause unnecessary collisions as the issue count grows.
- **Why it matters:** Users expect "Short IDs" to match the unique suffix of the hash part (e.g., `a1b` matching `bt-a1b2`). A broad suffix match on the entire ID (including the numeric parts of other IDs) will lead to frequent "ambiguous ID" errors even when a prefix-of-suffix match would have been unique.
- **Where:** `internal/issue.go` (lines 191-193).
- **Severity:** Medium

### 3. Archive Command Data Loss Risk
- **What's wrong:** The `archive` command removes issues from `issues.jsonl` first (using an atomic rename) and then appends them to `archive.jsonl`. If the process is interrupted or fails (e.g., disk full, permissions error) after the first step but before the second, the archived issues are lost from both files.
- **Why it matters:** While "recoverable from git" as noted in the comments, a core tool should not have a window where data is deleted from the source of truth before being safely committed to the destination.
- **Where:** `cmd/archive.go` (lines 74-95).
- **Severity:** Medium

### 4. Fragile Status Filtering in `ready` Command
- **What's wrong:** `cmd/ready.go` hardcodes the inclusion of only `open` and `in_progress` statuses. While this matches current documentation, it differs from `cmd/list.go` which uses `internal.TerminalStatuses`. If new non-terminal statuses are added (e.g., `draft` or `pinned` mentioned in `docs/ENUMS_AND_FORMATS.md`), they will be silently excluded from `bt ready`.
- **Why it matters:** Inconsistent filtering logic between `list` and `ready` makes the system harder to maintain and extend.
- **Where:** `cmd/ready.go` (lines 43-46).
- **Severity:** Low

### 5. Dependency on `USER` Environment Variable
- **What's wrong:** `cmd/create.go` and `cmd/dep.go` rely on `os.Getenv("USER")` to populate the `created_by` field.
- **Why it matters:** In environments where `USER` is not set (e.g., certain CI/CD pipelines, containers, or restricted shells), issues will be created with an empty `created_by` field, which might break downstream tools or lookups.
- **Where:** `cmd/create.go` (line 52), `cmd/dep.go` (line 81).
- **Severity:** Low

### 6. JSONL Line Length Limit
- **What's wrong:** `internal.LoadIssues` uses a fixed 1MB buffer for the `bufio.Scanner`.
- **Why it matters:** While 1MB is large for a typical issue, issues with very long descriptions or hundreds of comments (common in long-running projects) could exceed this limit, causing `bt` to fail to load the entire issues file.
- **Where:** `internal/issue.go` (line 90).
- **Severity:** Low

## Summary
The codebase is well-structured and follows Go idioms, with good attention to JSON preservation and basic concurrency. However, the divergence from the documented ID/Epic model and the broad ID resolution logic should be addressed to ensure long-term usability and compatibility with the intended design.
